# MyAI — AI 时代的个人能力资产飞轮

> **Archive Note**: 本文档是早期愿景探索，记录了项目从”配置同步工具”到”AI 资产飞轮”的思考过程。当前产品战略已 pivot 到团队 workflow portability 方向，具体见 [STRATEGY.md](../../STRATEGY.md)。本文档仅作为历史备忘保留，不再作为产品定义的权威来源。

> 一句话：把 prompts、MCPs、preferences、skills 从”散落在各个平台里的配置和聊天残留”变成”可积累、可迁移、可复用的个人资产”。

---

## 1. 背景判断

这个项目最初来自一个具体痛点：Claude Code 和 Codex CLI 的 MCP Server、Skills、个人指令分散在不同位置，无法共享。

但深入看，这不是单纯的“配置同步”问题，而是一个新的个人软件类别问题：

- AI 工具每天都在消耗用户的时间和思考
- 用户确实在持续产出高价值资产
- 这些资产被锁在不同平台、目录和账号里
- 一旦换工具、换机器、换平台，积累几乎不能继承

历史上，密码管理器解决的是“身份资产分散”，dotfiles 解决的是“开发环境资产分散”，Obsidian 解决的是“知识资产分散”。

MyAI 想解决的是：

**AI 时代，个人能力资产分散。**

---

## 2. 核心假设

项目成立的前提，不是“大家需要另一个 AI 工具”，而是下面这个假设成立：

**用户愿意把自己沉淀下来的 AI 使用经验，当作长期资产来保存、找回、迁移和复用。**

这些资产包括：

- `prompts`：反复验证过的高质量提示词
- `mcps`：连接外部能力的 MCP 配置
- `preferences`：个人风格、规则和长期偏好
- `skills`：可复用的半结构化工作流

如果这个假设成立，MyAI 就不是一个同步小工具，而是用户的个人 AI 能力仓库。

---

## 3. 产品定位

**MyAI 是一个本地优先的个人 AI 资产层。**

它不是某一个 AI 聊天产品，也不是只做某一种配置格式的迁移工具。它提供三层能力：

1. **沉淀**：把有价值的 AI 资产保存下来
2. **复用**：在同一工具或不同工具中重新找到并使用
3. **迁移**：换工具、换机器、换平台时不从零开始

长期看，MyAI 还应支持：

4. **组织**：资产越来越多时，自动分类、去重、推荐
5. **分发**：个人资产能同步、分享、协作、公开发布

---

## 4. 创业飞轮

项目不应该按“功能列表”推进，而应该按飞轮推进。

### 核心飞轮

1. 用户在某个 AI 工具里产出高价值资产
2. MyAI 以低摩擦方式把它保存为标准化文件
3. 用户稍后能搜索、找回、再次使用
4. 用户进一步在别的工具里复用这份资产
5. 用户感受到“积累没有丢”，于是更愿意继续沉淀
6. 资产规模增长后，组织、推荐、同步价值继续增强
7. 更强的使用价值反过来驱动更多保存行为

这个飞轮里最关键的不是“同步”本身，而是：

**保存之后，用户必须很快体会到复用价值。**

如果只有保存，没有复用，产品会退化为一个仓库。
如果只有同步，没有沉淀，产品会退化为一个配置搬运器。

---

## 5. 分阶段策略

愿景不缩，但启动顺序必须正确。

### 阶段一：资产沉淀飞轮

目标：先证明用户愿意存。

聚焦能力：

- 保存 prompt / MCP / preference
- 基础元数据和分类
- 列表、搜索、取回
- 本地 Git 仓库存储

目标用户：

- 多工具重度 AI 用户
- Claude Code / Codex / Cursor 用户
- 已经感受到“好东西总找不到”的开发者

must-win 指标：

- 用户在 7 天内发生重复保存
- 同一资产被二次使用
- 用户愿意把仓库作为长期目录维护

这个阶段解决的问题是：

**“值不值得存。”**

### 阶段二：跨工具复用飞轮

目标：证明“换工具不归零”。

聚焦能力：

- 从统一仓库同步到 Claude Code / Codex / Cursor
- 处理 MCP 配置格式差异
- 基础导入、导出、冲突提示
- Prompts 和 MCPs 优先，skills 作为兼容项

must-win 指标：

- 用户在两个以上工具之间复用同一资产
- 新机器初始化时间显著下降
- 至少一种同步场景形成高频习惯

这个阶段解决的问题是：

**“值不值得迁。”**

### 阶段三：智能组织飞轮

目标：让资产越多越值钱，而不是越多越乱。

聚焦能力：

- 自动分类、标签、去重
- usage metadata
- 相关推荐
- 从 prompt 升级为 skill 的建议

must-win 指标：

- 资产数增加后，检索命中率不下降
- 用户开始依赖历史资产，而不是每次临时现写

这个阶段解决的问题是：

**“值不值得持续积累。”**

### 阶段四：分发与协作飞轮

目标：从个人资产工具变成网络化基础设施。

聚焦能力：

- GitHub/private repo 同步
- 团队共享 vault
- 模板分发和公开分享
- 导入第三方资产库

must-win 指标：

- 用户开始分享、协作、fork 资产库
- 团队场景出现复用和共建行为

这个阶段解决的问题是：

**“值不值得传播。”**

---

## 6. 为什么从开发者切入

项目长期面向所有 AI 用户，但最早不该从“所有人”切。

开发者是更好的起点，因为他们：

- 天然使用多个 AI 工具
- 对 prompt、规则、配置的复用需求最强
- 能接受本地仓库、Git、CLI、MCP 这些概念
- 愿意容忍早期产品粗糙，只要核心价值足够强

所以冷启动应该围绕：

**多工具开发者的 prompts + MCPs + preferences。**

这不是缩小愿景，而是选择最先能把飞轮转起来的人群。

---

## 7. 当前阶段的产品重点

现阶段不该平均推进所有方向，而应围绕飞轮启动做取舍。

### 当前优先

- 一个清晰的本地资产模型
- 最少可用的保存 / 搜索 / 列出能力
- MCP 配置的跨工具适配
- Prompt 和 preference 的稳定读写

### 当前暂缓

- 复杂的 skills 跨工具统一
- 浏览器扩展
- 团队协作
- 公开 marketplace
- 过早做“大而全”的 AI 自动整理

原因很简单：

`skills` 在不同工具间的真实兼容性弱，而 `MCP + prompt + preference` 更接近用户的核心复用资产。

---

## 8. 北极星指标

这个项目不能只看安装量，要看飞轮有没有启动。

建议北极星指标：

**每周被再次使用的资产数（weekly reused assets）**

辅助指标：

- 每周新增保存资产数
- 7 日内重复保存用户占比
- 双工具及以上复用率
- 新环境恢复成功率
- 搜索后命中并使用的比例

如果“保存很多，但没有复用”，说明仓库在堆垃圾。
如果“复用很多，但新增很少”，说明沉淀动力不够。

---

## 9. 技术与产品约束

项目有几个硬约束：

- **本地优先**：用户资产必须能脱离 MyAI 独立存在
- **标准格式**：Markdown + YAML + 可读目录结构
- **Git 友好**：可版本化、可备份、可分享
- **AI Native**：管理方式优先是对话，不是配置编辑
- **不造封闭平台**：MyAI 不应该变成新的锁定层

技术上最现实的挑战仍然是 MCP 跨工具差异：

- JSON vs YAML
- object vs array 数据结构
- headers / streamable HTTP 支持差异
- 各工具变量语法不一致

这意味着跨工具同步必须是“适配器产品”，而不是简单复制文件。

---

## 10. 风险判断

### 风险一：概念太大，启动太散

如果一开始同时做 GPT、Claude Project、浏览器扩展、CLI、同步、组织、分享，团队会一直在铺摊子，飞轮不会转起来。

### 风险二：把“保存”误当成终局

用户不会为了存而存。必须尽快让用户感受到“以后还能用上”。

### 风险三：低估 skills 的非标准化

skills 有价值，但跨工具的通用部分很少，过早把它当成主轴，会拖慢整个产品。

### 风险四：目标用户漂移

大众用户规模大，但痛点浅；多工具开发者规模小，但问题真。冷启动必须先站稳前者还是后者，不能两头兼顾。

---

## 11. 现阶段结论

MyAI 的正确叙事不是：

“做一个统一管理 prompts、skills、MCP 的全能 AI 平台。”

而是：

**先把 AI 资产沉淀下来，再让它跨工具复用，再让它随着积累变得更有价值，最后形成分发与协作网络。**

这才是创业公司的推进方式：

- 第一阶段证明值得存
- 第二阶段证明值得迁
- 第三阶段证明越存越值钱
- 第四阶段证明值得传播

---

## 12. 接下来最应该定义的事

接下来最重要的不是补更多功能，而是把以下三件事写硬：

1. `v0.1` 的精确边界：哪些资产类型必须支持，哪些明确不做
2. 第一批目标用户画像：谁会高频保存并复用这些资产
3. 最小闭环：一次保存后，用户最短多久能在另一个场景里重新用上

如果这三件事定义清楚，后面的 CLI、MCP、Project、GPT、扩展，都只是不同的交付面。
